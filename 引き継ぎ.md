## 🚨 重要な決定事項・落とし穴

### 設計判断 (spec で確定済)
1. **トークンは Script Properties に格納、ハードコードしない** — GitHub リポジトリは Public のため
2. **物理削除しない、論理削除 (状態=キャンセル)** — 履歴保持・誤操作対策
3. **vehicle_delete のみ force 二段確認** — 予定で使用中の場合 force:true 必須
4. **スタッフ判定は 5 分キャッシュ** — マスタ更新後 5 分以内は旧結果が返る
5. **CORS 回避: PWA fetch の Content-Type は text/plain** — GAS は preflight 不対応のため
6. **GAS_URL は PWA に直接埋め込み** — ローカル開発+本番デプロイの切替なし、シンプル運用

### 落とし穴 (実装中に判明)
1. **新規 Apps Script プロジェクトはデフォルト Rhino**(旧)で作成される
   → `appsscript.json` に `"runtimeVersion": "V8"` 必須。これがないと `let`/`const`/`=>` で構文エラー
2. **clasp 3.x の `clasp deploy` は version + deployment を自動作成** — `clasp version` を別途実行する必要なし
3. **Container-bound script は Drive API では作成不可** — 「拡張機能 → Apps Script」のブラウザ操作 1 回だけ必須 (今回 Step ①)
4. **OAuth スコープ初回認可はブラウザ必須** — clasp deploy だけでは認可ダイアログ通せない (今回 Step ②)
5. **curl on Windows の POST が 411 エラー** — Content-Length ヘッダ不在で Google が拒否。Python urllib なら OK
6. **`Math.random()*1000` は 0-999 (可変長)、padStart なし** — Schedule ID 末尾が `S-...-7` のように短い場合あり。Vehicle/Staff の `padStart(3,'0')` とは非対称 (per-plan、意図的)
7. **getLastRow ベース ID は物理削除で衝突** — Vehicle/Staff ID 生成のみ。Schedule ID は timestamp ベースなので影響なし。Phase 2 で UUID 化検討
8. **`test_all` 関数は本番 Token 設定済みなら abort** — 安全装置。「ABORT ログが出れば認可は通った」と覚える
9. **PWA 初回起動時に SCHEDULE_TOKEN を localStorage に保存** — ユーザーに `prompt()` で入力させる。Phase 2 で別 UI に
10. **`<meta name="theme-color">` と manifest.json の theme_color は両方 #1D9E75** — iOS Safari と Android Chrome で別箇所に効く

### 既知の Phase 2 改善項目
- Vehicle ID 生成を UUID 化 (collision 防止)
- Staff キャッシュ invalidation 機構 (世代カウンタ方式)
- xlsx エクスポート (`exportSheetAsXlsxBase64_` 流用)
- お客様起点予約フロー (営業時間・枠数の運用 1ヶ月後の実績ベースで設計)
- LINE Push 通知 (前日リマインダー)
